# Finished Shelf - *17th April, 2026*

As nice as it is to have system calls to communicate from user to kernel, it is quite the pain to figure out exactly how to do that. I don't want system calls that do the shelf functions automatically for it. What I want are system calls that are general purpose and reusable in the future for other things, but at the same time, I somehow have to feed this output back to the shell in such a limited form.

I got through the hell of porting the kernel shell to the user shelf, and now it's *much* more nicer to get rid of the shell. Before, the shell was **part** of the kernel. Now, the shell is just another program named shelf on the hard disk. It feels *so* much better when it's like this, even if it's a tad-bit slower because of the system calls, it's actually worth it.

Of course, I'm going to leave the kernel shell in the source code, as well as the final `.bin` file, because I don't want to get rid of it. I have plans for the kernel shell in the future, but, for now, it can just stay in the kernel unused.

I did find writing these functions absolutely frustrating - especially `rm`, because I found out that the FAT32 functions change the given path string pointer around, so the shelf reads some random string after it. This was never a problem until now, but the remove command would sometimes remove, and sometimes fail. I don't even know what I did, but at some point, I compiled and ran the kernel, and it started working. Don't ask too much questions. Some of the commands were like this, the others were much more straightforward.

For `ls`, I wanted to implemented `opendir`, `readdir`, and `closedir`, but gave up half-way, because I kept going into page faults, and I was getting annoyed. I didn't need all these functions (yet), so I didn't do it, and made my own way to passing the data through, and that was *so* much easier.

This is too much software for two days, and I'm going to move back over into hardware stuff. I also kept losing my functions, so I wrote out a custom function in the make file... I should probably name it something, because it's to the point that it's no longer a "Make file", it kind of is a project manager. I'll think of a nice name later. I wrote a custom script to parse the entire source code and track where everything is, and make it easy for me to find these functions.

I also realise that this script will probably make it easy for anyone to contribute to the code, but I don't expect anyone to. Regardless, the `m-funcs`, as I have decided to call it, will help a lot. 'm' is also a very nice letter, it's a voiced bilabial nasal: starts with open mouth and ends closed, and that means any word after it feels perfect to say: `m defs`, `m help`, etc. Some do sound weird, but I like it enough to leave it at just 'm'.

I have no clue what to do now with the kernel, now that I have moved the shelf out. I really don't want to make graphics yet, because I assume it's going to be in the way of my flow. I prefer not designing and actually fixing the stuff no one can see before I get on with it. The Egyptians didn't build the pyramids from top to bottom, they probably started from below.
